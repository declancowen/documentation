## Separation of Concerns

The codebase follows a strict layered architecture:

| Layer | Responsibility | Location |
|-------|---------------|----------|
| UI | React components, presentation only | `components/` |
| Logic | Business logic via custom hooks | `hooks/` |
| Data | API calls and data fetching | `lib/api/` |
| State | Global state via Context providers | `context/` |

## Design Principles

<AccordionGroup>
  <Accordion title="API-First Security">
    All data mutations go through API routes (`/api/*`). The client never calls Supabase directly for writes. The Supabase service key is only used server-side. RLS policies act as the last line of defense.
  </Accordion>
  <Accordion title="Component Reusability">
    UI components live in `componentlibrary/` (shadcn/ui). Feature components are organized by domain: `auth/`, `recipe/`, `user/`. Check existing components before creating new ones.
  </Accordion>
  <Accordion title="Type Safety">
    Strict TypeScript throughout — no `any` types. Database types use `snake_case`, API responses use `camelCase`. Conversion utilities handle the mapping.
  </Accordion>
  <Accordion title="Shared Code">
    The `@recipe-room/shared` package provides types, Zod schemas, constants, and utilities used by both web and mobile apps.
  </Accordion>
</AccordionGroup>

## Data Flow

### Create Recipe

```
User Action → Form Validation (Zod) → API Route →
Supabase Insert → Response → React Query Cache Update →
Toast Notification → Redirect
```

### Image Upload

```
Select Image → Client Resize/Compress → POST /api/upload/image →
Validate Server-Side → Supabase Storage → Return URL →
Update Form State → Display Preview
```

### Follow User

```
Click Follow → Optimistic UI Update → API Route →
Supabase Insert → Confirm/Rollback → Update Follower Count
```
